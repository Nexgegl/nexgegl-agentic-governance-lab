# Competitor Trust Audit Skill

## 1. Purpose
يدقق ادعاءات منافس (موقع، مادة تسويقية، عرض مبيعات) مقابل عشرة محاور مصداقية موضوعية، وينتج حكماً واحداً: TRUST / VERIFY / AVOID.

## 2. When to Use
- قبل مقارنة NEXGEGL/ESTARED بمنافس في عرض مبيعات أو مادة تموضع.
- عند تقييم ادعاء منافس يبدو مبالغاً فيه أو غير مدعوم.
- عند فحص "شريك محتمل" أو مورّد تقني قبل تكامل.

## 3. Inputs Required

**Required:**
- مصدر ادعاءات المنافس (رابط، لقطة شاشة، نص تسويقي، عرض تقديمي).
- المحور المطلوب فحصه إن كان محدداً (أمني/تسعير/إلخ)، أو "الكل" لفحص شامل.

**Optional:**
- شهادات/مراجعات مستقلة عن المنافس إن توفرت.
- سياق تنافسي (هل هذا منافس مباشر لأي منتج NEXGEGL؟).

**Missing Input Handling:**
أي محور لا يوجد له دليل كافٍ في المصدر يُصنَّف "غير قابل للتحقق من المصدر المتاح" — لا يُفترض سلباً أو إيجاباً بلا دليل.

## 4. Operating Rules
- Do not invent facts. لا يُنسب للمنافس ادعاء أو ممارسة لم تظهر في المصدر المقدَّم.
- افصل بوضوح بين ما هو **ادعاء موثَّق من المصدر** وما هو **استنتاج/تخمين محلل**.
- إن كان الدليل غير كافٍ لأي محور، يُذكر ذلك صراحة بدل إصدار حكم متسرع.
- **Signal is not Decision.** هذا التدقيق إشارة تحليلية تدعم قراراً بشرياً (مثال: هل نستخدم هذه المقارنة في عرض مبيعات؟) — لا يُستخدم التدقيق نفسه كقرار نهائي بمعاملة المنافس.
- **لا قرار مؤسسي (مثل نشر مقارنة علنية أو اتهام منافس) بلا Evidence + Authority + Audit** — أي استخدام علني لنتائج هذا التدقيق يتطلب مراجعة قانونية/تسويقية بشرية قبل النشر.

## 5. Execution Workflow
1. اجمع كل الادعاءات القابلة للفحص من المصدر (اقتباس مباشر لكل ادعاء).
2. لكل محور من العشرة (§ المتطلبات أدناه)، ابحث عن دليل داعم أو نافٍ ضمن المصدر فقط.
3. صنّف كل محور: **موثَّق** / **غير موثَّق** / **غير قابل للتحقق من المصدر المتاح**.
4. اجمع "الأعلام الحمراء" (Red Flags) — تناقضات داخلية، أرقام غير قابلة للتصديق، غياب أي إثبات لادعاء جوهري.
5. اشتق حكماً إجمالياً واحداً بناءً على كثافة الأعلام الحمراء ونسبة الادعاءات غير الموثَّقة.
6. املأ القالب في §6.

## 6. Output Format

```
COMPETITOR TRUST AUDIT — [اسم المنافس/المصدر] — [التاريخ]

Claims: [قائمة الادعاءات الرئيسية كما وردت حرفياً]

Proof: [موثَّق / غير موثَّق / غير قابل للتحقق] لكل ادعاء رئيسي

Security Signals: [موثَّق / غير موثَّق / غير قابل للتحقق] — [التفاصيل]

Privacy/Data Handling: [موثَّق / غير موثَّق / غير قابل للتحقق] — [التفاصيل]

Fake Metrics Risk: [منخفض/متوسط/مرتفع] — [السبب]

Testimonials Credibility: [موثَّق/غير موثَّق/غير قابل للتحقق] — [التفاصيل]

Integration Claims: [موثَّق/غير موثَّق/غير قابل للتحقق] — [التفاصيل]

Pricing Transparency: [شفاف/غير شفاف/غير قابل للتحقق] — [التفاصيل]

Execution Feasibility: [واقعي/مشكوك فيه/غير قابل للتقييم] — [التفاصيل]

Red Flags:
- [علم أحمر 1]
- [علم أحمر 2]

VERDICT: TRUST / VERIFY / AVOID
مبرر الحكم: [جملتان توضحان سبب الحكم استناداً للمحاور أعلاه]
```

## 7. Quality Gates

| الحالة | معيار قابل للملاحظة |
|---|---|
| **PASS** (تدقيق مقبول للاستخدام الداخلي) | كل محور من العشرة مصنَّف بأحد الخيارات الثلاثة مع دليل أو سبب واضح، والحكم النهائي متسق منطقياً مع المحاور. |
| **FIX** | محور واحد أو أكثر بلا تصنيف، أو الحكم النهائي غير متسق مع تفاصيل المحاور (مثال: حكم TRUST رغم 4 أعلام حمراء موثقة). |
| **FAIL** | التدقيق ينسب ادعاءً أو ممارسة للمنافس غير موجودة في المصدر، أو يصدر حكم AVOID/TRUST دون أي دليل داعم مذكور. |

## 8. Anti-Patterns
- إصدار حكم AVOID بدافع تحيز تنافسي دون أعلام حمراء موثقة فعلياً في المصدر.
- الخلط بين "غير قابل للتحقق" و"غير موثَّق" (الأول يعني غياب معلومة، الثاني يعني ادعاء بلا دليم رغم إمكانية التحقق).
- استخدام نتيجة هذا التدقيق مباشرة في مادة تسويقية علنية دون مراجعة بشرية/قانونية.
- تضخيم Red Flags بصياغة اتهامية بدل وصف محايد للفجوة.

## 9. Example Prompt
"استخدم Competitor Trust Audit Skill لفحص صفحة التسعير والمزايا الأمنية لمنافس [X] بناءً على لقطات الشاشة المرفقة، وأصدر حكماً TRUST/VERIFY/AVOID."
