# Evidence Pack Builder Skill

## 1. Purpose
يجمّع ويقيّم حزمة الأدلة (Evidence Pack) المطلوبة لدعم قرار مؤسسي معيّن، ويحدد مدى جاهزية هذا القرار للاعتماد بناءً على اكتمال الدليل.

## 2. When to Use
- قبل أي قرار مؤسسي يتطلب توثيق Evidence + Authority + Audit وفق المعيار الرئيسي.
- عند إعداد ملف لمراجعة تدقيق (Audit) داخلي أو خارجي.
- عند الشك فيما إذا كان الدليل المتوفر كافياً لاتخاذ قرار بعينه.

## 3. Inputs Required

**Required:**
- القرار المؤسسي المستهدف (ما الذي يُراد اعتماده فعلياً).
- كل المستندات/البيانات المتاحة حالياً ذات الصلة.

**Optional:**
- سياسة حوكمة داخلية تحدد الحد الأدنى من الأدلة المطلوبة لهذا النوع من القرارات، إن وُجدت.
- تاريخ قرارات مشابهة سابقة ومستنداتها.

**Missing Input Handling:**
أي دليل مطلوب غير متوفر يُدرَج صراحة تحت "Evidence Gaps" مع تحديد أثره على جاهزية القرار — لا يُفترض وجوده أو استبداله باستنتاج.

## 4. Operating Rules
- Do not invent facts. لا تُدرَج وثيقة أو مصدر كـ"متوفر" ما لم يكن فعلياً مُقدَّماً أو مُتحقَّقاً منه.
- افصل بين **الدليل المستلم فعلياً (Evidence Received)** و**الدليل المطلوب نظرياً (Evidence Required)** بوضوح تام.
- قيّم **موثوقية المصدر (Source Reliability)** لكل دليل — دليل من مصدر أولي (نظام مالي مباشر) أعلى موثوقية من ادعاء لفظي غير موثَّق.
- **لا قرار مؤسسي (Decision) بلا Evidence + Authority + Audit** — هذا الـ Skill هو الأداة المباشرة لفحص هذا الشرط قبل أي اعتماد.
- **Signal is not Decision.** اكتمال حزمة الأدلة شرط ضروري لكن غير كافٍ وحده — يبقى القرار فعلاً بشرياً يتطلب صلاحية موثقة بعد اكتمال الدليل.

## 5. Execution Workflow
1. حدد بدقة **القرار المستهدف** الذي يُراد بناء حزمة الأدلة له.
2. اشتق **Evidence Required**: ما الحد الأدنى من الأدلة اللازمة لهذا النوع من القرار (استناداً لسياسة داخلية إن وُجدت، أو للمنطق المؤسسي العام: مصدر بيانات + تفويض + سجل).
3. سجّل **Evidence Received**: ما هو متوفر فعلياً الآن من هذه القائمة.
4. اشتق **Evidence Gaps** بالفرق بين المطلوب والمستلم.
5. قيّم **Source Reliability** لكل دليل مستلم (أولي موثَّق / ثانوي / لفظي غير موثَّق).
6. اشتق **Decision Readiness**: هل الحزمة كافية للاعتماد الآن، أم لا؟
7. حدد **Audit Trail Requirement**: كيف سيُوثَّق هذا القرار بعد اعتماده لضمان قابلية المراجعة لاحقاً؟
8. إن لم تكن الحزمة كاملة، حدد **Next Evidence Request** بدقة: ما الدليل التالي الذي يجب طلبه، ومن.

## 6. Output Format

```
EVIDENCE PACK — [القرار المستهدف] — [التاريخ]

Evidence Required:
- [بند 1]
- [بند 2]

Evidence Received:
- [بند 1: مصدره]
- [بند 2: مصدره، أو "غير مستلم"]

Evidence Gaps:
- [الفجوة 1]
- [الفجوة 2، أو "لا فجوات"]

Source Reliability: [لكل دليل مستلم: أولي/ثانوي/لفظي غير موثَّق]

Decision Readiness: [جاهز للاعتماد / جاهز جزئياً / غير جاهز] — [المبرر]

Audit Trail Requirement: [كيف سيُوثَّق القرار وأين يُحفظ الأثر]

Next Evidence Request: [الدليل التالي المطلوب، من الجهة، وبأي مهلة إن وُجدت]
```

## 7. Quality Gates

| الحالة | معيار قابل للملاحظة |
|---|---|
| **PASS** (حزمة جاهزة للعرض على صاحب الصلاحية) | Evidence Required وReceived وGaps مفصولة بوضوح تام، كل دليل مستلم مقيَّم بموثوقيته، Decision Readiness مبرر منطقياً بالفجوات المذكورة. |
| **FIX** | فجوة مذكورة دون تحديد Next Evidence Request مقابلها، أو تقييم موثوقية مصدر مفقود لدليل مستلم. |
| **FAIL** | الحزمة تصنَّف "جاهزة للاعتماد" رغم وجود فجوة حرجة في Evidence Gaps، أو تُقدَّم كأنها القرار نفسه بدل كونها تمهيداً له. |

## 8. Anti-Patterns
- تصنيف دليل لفظي غير موثَّق كـ"أولي" لتسريع Decision Readiness.
- إغفال Audit Trail Requirement بحجة أن "القرار بسيط".
- الخلط بين Evidence Required (نظري) وEvidence Received (فعلي) في قائمة واحدة غير مفصولة.
- الاستنتاج بأن غياب اعتراض يعني اكتمال الدليل.

## 9. Example Prompt
"استخدم Evidence Pack Builder Skill لتقييم جاهزية قرار إيقاف خدمة لعميل متأخر السداد، بناءً على المستندات المتاحة حالياً (فاتورة، سجل تواصل، لا يوجد عقد موقَّع محدَّث)."

```
EVIDENCE PACK — [القرار] — [YYYY-MM-DD]
Evidence Required: [قائمة]
Evidence Received: [قائمة + مصدر كل بند]
Evidence Gaps: [قائمة أو "لا فجوات"]
Source Reliability: [تقييم لكل بند]
Decision Readiness: [حالة + مبرر]
Audit Trail Requirement: [الوصف]
Next Evidence Request: [التفاصيل أو "لا يوجد — الحزمة مكتملة"]
```
