# Client Discovery Skill

## 1. Purpose
يحوّل طلب عميل خام وغير منظَّم (رسالة، اجتماع، بريد) إلى نطاق اكتشاف (Discovery Scope) منظَّم يمهّد لعرض تسعير أو مقترح تقني.

## 2. When to Use
- عند استلام أول طلب أو استفسار من عميل محتمل، قبل أي عرض سعر.
- عند الحاجة لتوثيق نطاق مشروع بشكل منظَّم قبل بدء أي تطوير.
- **لا يُستخدم** لإنتاج عرض سعر نهائي — ذلك دور `pricing-scope-skill.md` الذي يُستخدَم بعد هذا الـ Skill.

## 3. Inputs Required

**Required:**
- نص أو ملخص طلب العميل كما ورد فعلياً.
- أي معلومات متاحة عن قطاع العميل ونوع عمله.

**Optional:**
- اجتماعات سابقة أو ملاحظات إضافية.
- أنظمة تقنية معروفة يستخدمها العميل حالياً.

**Missing Input Handling:**
أي بند من بنود النطاق (§5) لا يوجد له معلومة كافية في طلب العميل يُدرَج صراحة تحت "Questions Before Proposal" — لا يُملأ بافتراض عام.

## 4. Operating Rules
- Do not invent facts. لا يُفترض نظام تقني أو عملية عمل للعميل لم تُذكر فعلياً في طلبه.
- افصل بين ما **ذكره العميل صراحة** وما هو **استنتاج محلل بناءً على قطاع العمل** (يُوسَم كـ `[استنتاج]`).
- إن كانت معلومة حاسمة لتحديد النطاق غير متوفرة، تُدرَج كسؤال قبل المقترح — لا تُخمَّن.
- **Signal is not Decision.** هذا الاكتشاف إشارة أولية لتشكيل مقترح، وليس التزاماً تعاقدياً نهائياً بأي نطاق أو سعر.
- **لا التزام تعاقدي (Decision) بلا Evidence + Authority + Audit** — أي نطاق ناتج هنا يمر لاحقاً عبر تسعير ومراجعة بشرية قبل أن يصبح عرضاً ملزماً.

## 5. Execution Workflow
1. استخرج **Problem Statement**: ما المشكلة الفعلية التي يحاول العميل حلها، بكلماته لا بتأويل تقني مبكر.
2. حدد **Users/Stakeholders**: من سيستخدم الحل فعلياً؟
3. حدد **Departments/Entities**: أي أقسام أو جهات داخل مؤسسة العميل معنية؟
4. حدد **Current Systems**: ما الأنظمة القائمة حالياً (إن ذُكرت)؟
5. حدد **Data Sources**: من أين تأتي البيانات التي سيتعامل معها الحل؟
6. حدد **Workflow States**: ما مراحل سير العمل الحالية أو المطلوبة؟
7. حدد **Approval Points**: أين تحدث نقاط موافقة/قرار بشرية في هذا السير؟
8. حدد **Dashboard Needs**: هل يحتاج العميل عرض بيانات/تقارير؟ ما شكلها؟
9. حدد **Risks**: أي مخاطر تقنية أو تنظيمية أو زمنية واضحة من الطلب.
10. حدد **MVP Scope**: أصغر نطاق ممكن يحل جوهر المشكلة.
11. حدد **Exclusions**: ما الذي لن يكون ضمن هذا النطاق صراحة، لتفادي توسع لاحق غير محكوم.
12. اجمع **Questions Before Proposal**: كل فجوة معلومات لم تُحسَم.

## 6. Output Format

```
CLIENT DISCOVERY — [اسم العميل/المشروع] — [التاريخ]

Problem Statement: [وصف المشكلة بكلمات العميل]

Users/Stakeholders: [قائمة]

Departments/Entities: [قائمة]

Current Systems: [قائمة أو "غير مذكور — راجع الأسئلة"]

Data Sources: [قائمة أو "غير مذكور"]

Workflow States: [قائمة المراحل]

Approval Points: [قائمة نقاط القرار البشرية]

Dashboard Needs: [وصف أو "غير مذكور"]

Risks: [قائمة]

MVP Scope: [وصف أصغر نطاق قابل للتسليم]

Exclusions: [قائمة صريحة لما هو خارج النطاق]

Questions Before Proposal:
- [سؤال 1]
- [سؤال 2]
```

## 7. Quality Gates

| الحالة | معيار قابل للملاحظة |
|---|---|
| **PASS** (جاهز لبدء التسعير) | كل بند من الـ 11 بنداً مملوء بمعلومة فعلية أو مُحال صراحة لقسم الأسئلة، Exclusions محددة بوضوح، MVP Scope أصغر من الطلب الكامل للعميل. |
| **FIX** | بند واحد أو أكثر فارغ دون إحالة لقسم الأسئلة، أو Exclusions غير مذكورة إطلاقاً. |
| **FAIL** | النطاق يخلط استنتاجات المحلل بحقائق العميل دون وسم، أو يقدَّم كالتزام تعاقدي نهائي بدل نطاق اكتشاف أولي. |

## 8. Anti-Patterns
- افتراض نظام تقني (مثال: "يستخدمون Salesforce على الأرجح") دون ذكر فعلي من العميل، ودون وسمه كاستنتاج.
- تضخيم MVP Scope ليشمل كل ما ذكره العميل بدل تحديد الأصغر الكافي.
- حذف قسم Exclusions بحجة "سيُحدَّد لاحقاً في العقد".
- تقديم هذا الاكتشاف كأنه عرض مُلزم للعميل.

## 9. Example Prompt
