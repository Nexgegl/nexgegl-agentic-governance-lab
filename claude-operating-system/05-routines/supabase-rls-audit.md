# Routine: تدقيق صلاحيات Supabase RLS
## Supabase RLS Audit — v1.0

> يُقرأ مع `03-sub-agents/security-rls-auditor.md`.

---

## 1. الغرض (Purpose)

تدقيق دوري ومنهجي لسياسات Row-Level Security (RLS) في أي مستودع NEXGEGL يستخدم Supabase/PostgreSQL، للتأكد من أن الوصول إلى البيانات محكوم بدقة ولا يوجد تسريب محتمل.

## 2. التفعيل (Trigger)

- دوري (مثال: شهرياً، أو حسب حساسية المستودع).
- إلزامي فوري عند أي تغيير في مخطط قاعدة البيانات (schema) أو سياسات RLS.
- عند انضمام مستودع جديد يستخدم Supabase إلى المنظومة.

## 3. ما يُفحص

1. **كل جدول يحتوي بيانات حساسة له سياسة RLS مفعّلة.** لا استثناءات دون تبرير موثَّق ومعتمد.
2. **السياسات تُطبَّق فعلياً حسب هوية المستخدم/الدور**، لا حسب افتراضات ضمنية قد تُكسر بتغيير بسيط في الكود.
3. **لا صلاحيات service_role مكشوفة من جهة العميل (client-side).**
4. **لا مفاتيح أو أسرار Supabase مُضمَّنة في الكود أو التوثيق العام.**
5. **مسارات API/Edge Functions تحترم نفس حدود RLS** ولا تلتف حولها عبر استخدام صلاحيات أعلى دون داعٍ.
6. **بيانات العملاء (Client Repos) معزولة منطقياً** بحيث لا يستطيع عميل الوصول لبيانات عميل آخر.

## 4. الوكيل المسؤول

**`security-rls-auditor`** (`03-sub-agents/security-rls-auditor.md`) — قراءة فقط، لا تعديل مباشر على سياسات إنتاج.

## 6. قاعدة حرجة

اكتشاف جدول حساس بلا سياسة RLS، أو سر مكشوف علناً، يُصعَّد **فوراً** لصاحب الصلاحية الأمنية البشرية دون انتظار الجدولة الدورية التالية — بصرف النظر عن موعد التقرير القادم.
